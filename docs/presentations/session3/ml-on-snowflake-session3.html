💡 The Challenge

Why Run Training Remotely?

Local notebooks are great for prototyping. Production training needs more.

💻

Local Training Limits

Limited RAM and CPU. Data must be downloaded. No GPUs. Results stuck on your laptop. Notebook crashes lose hours of work.

⚡

ML Jobs on SPCS

Elastic compute (GPU/CPU). Data stays in Snowflake — zero egress. Auto-session inside containers. Scale from 1 to N nodes with one parameter.

🔒

Governance & Security

Training runs inside Snowflake’s security perimeter. Same RBAC as your data. Full audit trail. No data copies to external services.

🔁

Reproducibility

Consistent container runtime. Pinned dependencies via requirements.txt or Dockerfile. Same environment every time.

🐳 Custom Runtime Environment

The runtime environment can be pre-compiled as a Docker image, giving you full flexibility to customize the OS, Python version, system packages, and libraries for your workload.

🔒 Internal Package Repos

Need to pull from Artifactory, Nexus, or a corporate PyPI mirror? Create an External Access Integration with a network rule allowing egress to your internal repo. Containers on the pool can then pip install from your private registry.

🚀 Overview

What Are ML Jobs?

Run your existing Python scripts on Snowflake compute pools — no infrastructure management, no data movement.

The Concept

ML Jobs let you submit Python scripts to run on Snowflake’s managed compute pools (CPU or GPU). Snowflake handles provisioning, dependency installation, execution, and teardown. Think of it as serverless ML compute — submit code, get results.

Submission Methods

submit_file()Single script, no local imports
submit_directory()Multi-file project with shared modules
@remoteDecorator for quick single-function jobs

submit_job.py
from snowflake.ml.jobs import submit_directory

job = submit_directory(
    dir_path="job_payload",
    entrypoint="train_simple.py",
    compute_pool="ML_DEMO_COMPUTE_POOL",
    stage_name="@DB.SCHEMA.JOB_PAYLOADS",
    target_instances=1,
    env_vars={
        "DATABASE": "ML_DEMO_PIPELINE_DB",
        "SCHEMA": "HEALTHCARE",
        "MODEL_NAME": "PATIENT_RISK_MODEL",
        "N_ESTIMATORS": "150",
    },
)

# Monitor the job
print(f"Job ID: {job.id}")
job.wait()  # blocks until done
print(job.get_logs())

Custom Docker Images

Full control over runtime. Exact reproducibility. Faster cold starts.

🐳 Docker

Two Approaches: requirements.txt vs Dockerfile

Choose based on your project’s needs. Start simple, upgrade when needed.

📜

requirements.txt

✓Zero setup — just list packages ✓Auto-installed at job start ✓No Docker knowledge needed ●~30–60s cold start (pip install) ✗No system packages (apt-get)
scikit-learn
xgboost
pandas
🐳

Dockerfile

✓Full OS control (apt-get, CUDA) ✓Exact reproducibility (frozen layers) ✓~5–10s cold start (image pull) ✓Pre-built wheels & multi-stage builds ●Requires Docker build & push step
FROM python:3.10-slim
RUN pip install snowflake-ml-python xgboost
COPY train.py /app/
ENTRYPOINT ["python", "train.py"]

Recommendation: Start with requirements.txt for development. Upgrade to Dockerfile for production when you need system packages or faster cold starts.

📦 Frameworks

Built-In Frameworks for Common Models

Snowflake wraps popular ML libraries with high-level APIs — handling distribution, data loading, and model registration for you.

XGBEstimator

Native distributed XGBoost. Auto-partitions data across workers via DataConnector. Supports CPU and GPU.

Data loadingAutomatic (DataConnector)
ScalingXGBScalingConfig
Outputxgb.Booster

LightGBMEstimator

Same pattern as XGBEstimator but for LightGBM. Handles distributed tree building natively.

Data loadingAutomatic (DataConnector)
ScalingLGBMScalingConfig
Outputlgbm.Booster

PyTorchDistributor

Wraps PyTorch DDP for multi-GPU training. Uses ShardedDataConnector for distributed data loading across workers.

StrategyDistributedDataParallel
Best forDeep learning / GPU
Outputtorch model state_dict

ManyModelTraining

Train one model per data partition (e.g. per region). Snowflake distributes partitions across workers and serializes each model.

Pattern1 model per partition key
SerializationAuto (cloudpickle)
Use casePer-store / per-region models

Why use these? You write training logic. Snowflake handles cluster setup, data distribution, fault tolerance, and model collection — all on your compute pool.

DataConnector

Efficient bridge from Snowflake tables to distributed workers. Auto-partitions and streams data — no .to_pandas() needed.

distributed_xgb.py (runs inside ML Job)
from snowflake.ml.modeling.distributors.xgboost import (
    XGBEstimator, XGBScalingConfig
)
from snowflake.ml.data.data_connector import DataConnector

# Efficient data loading (auto-partitions)
train = DataConnector.from_dataframe(
    session.table("TRAINING_FEATURES"))
eval = DataConnector.from_dataframe(
    session.table("TEST_FEATURES"))

estimator = XGBEstimator(
    params={
        "objective": "multi:softmax",
        "num_class": 4,
        "max_depth": 8,
        "learning_rate": 0.1,
    },
    scaling_config=XGBScalingConfig(
        num_workers=-1,        # auto-detect
        use_gpu=None,          # auto-detect
    ),
)

booster = estimator.fit(
    dataset=train, input_cols=input_cols,
    label_col="RISK_LEVEL", eval_set=eval)
