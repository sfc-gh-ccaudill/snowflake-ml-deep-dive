Part 1
Inference

Two paths to predictions: SQL in the warehouse, REST in the compute pool.

⚡ Concept

What is Inference?

Passing new data through a trained model to get a prediction. The model applies learned patterns to produce actionable output.

⚡ Inference

Two Inference Paths

SQL inference runs on your virtual warehouse. Real-time inference runs on the SPCS compute pool. Same model, different compute.

💻 Code

Inference Code Examples

Call the same model two ways: HTTP for real-time apps (left), SQL for batch pipelines (right).

🌐 External apps call the REST endpoint via the Gateway
REST API (Compute Pool)
import json, requests, pandas as pd

df = pd.DataFrame({
    "AGE": [65], "BMI": [28.5],
    "HEART_RATE": [88], ...
})

payload = {"dataframe_split":
  json.loads(df.to_json(orient="split"))}

url = "https://<gw>.snowflakecomputing.app/predict"
headers = {
  "Authorization": f'Snowflake Token="{PAT}"',
  "Content-Type": "application/json",
}

response = requests.post(
    url, json=payload, headers=headers)
result = response.json()
# {"data":[[0,{"output_feature_0":"HIGH"}]]}
📊 SQL queries reference the model directly — no service needed
SQL Inference (Warehouse)
-- Default version (simplest)
SELECT PATIENT_ID,
  PATIENT_RISK_MODEL!PREDICT(
    AGE, BMI, HEART_RATE, ...
  ) AS prediction
FROM RAW_PATIENT_DATA;

-- Specific version via alias
WITH v AS MODEL PATIENT_RISK_MODEL
         VERSION production
SELECT PATIENT_ID,
  v!PREDICT(AGE, BMI, ...) AS prediction
FROM RAW_PATIENT_DATA;

-- Batch scoring (CTAS)
CREATE TABLE PREDICTIONS AS
SELECT *, PATIENT_RISK_MODEL!PREDICT(...)
       AS prediction
FROM RAW_PATIENT_DATA;

Part 2
Model Deployment

From the Model Registry to a managed REST endpoint — one API call.

🚀 Architecture

Model Serving Architecture

A registered model version is deployed as a managed container service within Snowflake’s security perimeter, exposed to applications via a stable REST endpoint.

📈 Scale

Service Scalability

Inference services scale horizontally with multiple instances and support GPU acceleration for compute-intensive models.

💻 Code

Deploying a Model Service

Key Parameters
service_compute_poolGPU or CPU pool
ingress_enabledPublic HTTP endpoint
autocaptureLog all requests
max_instancesHorizontal scaling
gpu_requestsGPU allocation
Important

autocapture is immutable — cannot be toggled after service creation. Must recreate the service.

deploy_model.py
from snowflake.ml.registry import Registry

reg = Registry(session, database_name="ML_DEMO_PIPELINE_DB",
               schema_name="HEALTHCARE")

# Get specific model version
mv = reg.get_model("PATIENT_RISK_MODEL").version("V1")

# Deploy as managed SPCS service
mv.create_service(
    service_name="PATIENT_RISK_SERVICE",
    service_compute_pool="ML_DEMO_COMPUTE_POOL",
    ingress_enabled=True,
    max_instances=3,
    autocapture=True,
)

# Get endpoint URL
services = mv.list_services()
endpoint = services['inference_endpoint'].iloc[0]

Part 3
MLOps

Model promotion, monitoring, drift detection, and automated retraining.

📚 Lifecycle

Model Promotion with Aliases

Aliases are exclusive labels that track lifecycle stages. Only one version can hold a given alias. Use the default version for zero-change SQL inference.

🔧 Production

Promoting Models for Real-Time Services

For inference services, promote new model versions by deploying a new service behind the gateway and gradually migrating traffic. Zero-downtime canary deployments.

📈 Observability

Auto-Capture & Drift Detection

Every inference is logged. Compare production distributions against baseline to detect when the world has changed.

📊 Monitor

Monitoring Architecture & Drift Types

Data Drift (Covariate Shift)

Input feature distributions change. e.g., patients are older, higher BMI than training data.

Prediction Drift

Output distribution shifts. e.g., model predicts “HIGH” more often than during training.

Concept Drift

The relationship between inputs and outputs changes. Same features now map to different risk levels.

Available Metrics
PSI KL_DIVERGENCE ACCURACY F1 COUNT
⚡ Automation

Automated Retraining with Alerts

SQL — Create drift alert
CREATE OR REPLACE ALERT
  PREDICTION_DRIFT_ALERT
  WAREHOUSE = ML_DEMO_WAREHOUSE
  SCHEDULE = 'USING CRON 0 6 * * *
    America/Los_Angeles'
  IF (EXISTS (
    SELECT 1
    FROM TABLE(
      MODEL_MONITOR_DRIFT_METRIC(
        'PATIENT_RISK_MONITOR',
        'PSI','RISK_LEVEL','DAY',
        DATEADD('day',-1,
          CURRENT_TIMESTAMP())
          ::TIMESTAMP_NTZ,
        CURRENT_TIMESTAMP()
          ::TIMESTAMP_NTZ))
    WHERE "value" > 0.25
  ))
  THEN
    EXECUTE TASK
      PIPELINE_FEATURE_ENG_TASK;

ALTER ALERT PREDICTION_DRIFT_ALERT
  RESUME;
